A phisher sends lure messages to phishing targets. Depending on their susceptibility, targets either trust or distrust the message. If they trust it, they visit the hook webpage and if this page does not appear scammy to them, they provide the requested information, which is then exploited by the phisher.

You can inspect the model's OESjs code on the OES GitHub repo.

Conceptual Model

A conceptual model, also called domain model, describes the real-world system under investigation by identifying the relevant types of objects and events, and describing their dynamics, allowing to understand what's going on in the real system. It does not describe any software/design artifact.

Conceptual Information Model

A conceptual information model describes the subject matter vocabulary used, e.g., in the system narrative, in a semi-formal way. Such a vocabulary essentially consists of names for

  • types, corresponding to classes in OO modeling, or unary predicates in formal logic,
  • attributes corresponding to binary predicates in formal logic,
  • associations corresponding to n-ary predicates (with n > 1) in formal logic.

The main categories of types are object types, agent types and event types. A simple form of conceptual information model is obtained by providing a list of each of them, while a more elaborated model, preferably in the form of a UML class diagram, also defines attributes and associations, including the participation of objects (of certain types) in events (of certain types).

The potentially relevant object types are: webpages, while the potentially relevant agent types are: phishers and phishing targets.

Potentially relevant types of events are:

  1. start of day: the start of a new business day for a phisher,
  2. send lure message: a phisher sends a lure message to many targets,
  3. receive lure message: a target receives a lure message from a phisher,
  4. read lure message header: a target reads the header data (sender and subject) of a lure message,
  5. open lure message body: a target opens a lure message for being able to read it,
  6. read lure message body: a target reads a lure message,
  7. visit hook page: a phishing target that trusts a lure message visits a hook webpage,
  8. look at hook page: a phishing target looks at a hook page,
  9. provide exploitable data: a phishing target trusts a hook page and provides the requested data,
  10. perceive exploitable data: a phisher notes that exploitable data has been provided by a target,
  11. exploit scammed data: a phisher exploits the data he has scammed from a target.

Both object types and event types, together with their participation associations, can be visually described in a conceptual information model in the form of a UML class diagram, as shown below.

Implementing the Information Design Model

For implementing the information design model, we have to code all object types and event types specified in the model in the form of classes.

The agent class ... can be coded with OESjs-Core4 in the following way:

class ... extends aGENT {
}

All agent classes inherit an id attribute and a name attribute from the pre-defined OES foundation class aGENT.
